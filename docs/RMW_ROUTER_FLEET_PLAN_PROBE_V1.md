# RMW Router Fleet Plan Probe V1

This artifact verifies the first C++ RMW/router actuation of fleet optimizer path IDs.

- Summary: `results_rmw_socket/docker_router_fleet_plan_probe_summary.json`
- Schema: `fleetrmw.rmw_router_fleet_plan_probe.v1`
- Policy: `FLEETQOX_RMW_PEER_POLICY=fleet_plan`
- Initial path plan: `/fleetqox/router_fleet_plan_probe=primary_wifi`
- Online-planner update: `/fleetqox/router_fleet_plan_probe=backup_5g+primary_wifi`
- Plan source: `FLEETQOX_RMW_FLEET_PATH_PLAN_FILE`

## Interpretation

- `FLEETQOX_RMW_PEERS` now accepts path-labeled peers such as `primary_wifi=router:port`.
- `FLEETQOX_RMW_FLEET_PATH_PLAN_FILE` lets the C++ RMW refresh topic-to-path rules from a controller-written file.
- In this probe, the file content is generated by `OnlineFleetPathPlanner`, which smooths per-path observations and calls `FleetQoEPathOptimizer`; it is not hand-written as a static route.
- The publisher starts with a primary-only plan, updates the plan file after the first publish, and the same live RMW process reloads the file for the next two publishes.
- In `fleet_plan` mode, `rmw_fleetqox_cpp` routes data frames to the selected path peers before falling back to older adaptive/all-peer behavior.
- This is the first C++ bridge from online planner path IDs to live Docker router peers. The remaining research work is feeding real router/subscriber measurements into that file-backed plan during a long-running ROS 2 session.
