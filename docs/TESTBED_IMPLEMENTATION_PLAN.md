# Testbed Implementation Plan

## Current Status

Implemented now:

- T0 FleetQoX simulator;
- experiment manifest;
- local runner for T0 scenarios;
- trace exporter for ns-3/OMNeT++ input;
- common result fields for early metrics.

Planned but not executed in this environment:

- ROS 2 `performance_test` runner;
- full `tc netem`/Docker run if Docker daemon is not active;
- ns-3/OMNeT++ trace-driven runner;
- Gazebo/Nav2 runner;
- Open-RMF runner.

## Practical Tool Choice

### First Practical Milestone

Use Docker containers or Linux network namespaces to represent robot hosts:

```text
robot_001 container
robot_002 container
...
operator container
fleet_router container
```

Apply `tc netem` or Pumba to container interfaces. Run ROS 2
`performance_test` inside containers with different RMWs.

This provides real ROS 2 networking without requiring a full robot simulator.

### Second Practical Milestone

Add Gazebo/Nav2 multi-robot simulation for a smaller number of robots:

```text
10-30 simulated robots with Nav2
network load generated by synthetic ROS nodes
operator-visible video/debug flows emulated or generated
```

Physics simulation does not need to scale to hundreds of robots initially. The
communication stress can be generated synthetically while a smaller set of
robots validates task impact.

### Third Practical Milestone

Use Open-RMF demos and synthetic fleet adapters:

```text
many synthetic robots
few full simulated robots
RMF task dispatch
communication impairment around adapters
```

This is more scalable than simulating hundreds of full robots in physics.

## Why Synthetic + Simulated Hybrid Is Necessary

Full high-fidelity simulation of 300 robots is expensive and often tells less
about middleware than a controlled synthetic graph. A hybrid testbed is more
credible:

- synthetic nodes scale traffic and discovery;
- a few simulated robots prove autonomy impact;
- fleet adapters prove task-level behavior;
- network emulators impose reproducible failure modes.

## Runner Design

All experiments should be manifest-driven:

```text
experiments/testbed_manifest.json
  -> scripts/run_suite.py
  -> results/*.jsonl
  -> notebooks/plots later
```

Each experiment has:

- `tier`;
- `name`;
- `purpose`;
- `baselines`;
- `scale`;
- `impairments`;
- `metrics`;
- `runner`.

The runner should skip unavailable external tools instead of failing the whole
suite.

## Network Simulator Runner

The trace exporter is available now:

```bash
python3 -m scripts.export_traces --scenario warehouse_100_constrained
python3 -m scripts.export_traces --include-non-sent
python3 -m scripts.run_suite --tier T0 --run --export-traces
```

The T1 environment probe is also available:

```bash
python3 -m scripts.run_t1_ros2_perf
python3 -m scripts.run_t1_ros2_perf --json
python3 -m scripts.run_t1_ros2_perf --plan-commands
```

On this machine, the runner may report missing ROS 2 tools. That is expected
unless the shell has sourced a ROS 2 installation with the `performance_test`
package available.

A Docker build target for T1 is available under `external/ros2-perf`:

```bash
docker compose -f external/ros2-perf/docker-compose.yml build
docker compose -f external/ros2-perf/docker-compose.yml run --rm ros2-perf
```

It builds Apex.AI `performance_test` with the ROS 2 plugin and prints the
FleetQoX command plan or runs the T1 smoke benchmark, depending on `T1_ARGS`.

Examples:

```bash
T1_ARGS="--plan-commands" docker compose -f external/ros2-perf/docker-compose.yml run --rm ros2-perf
T1_ARGS="--scenario small_messages_many_endpoints --run" docker compose -f external/ros2-perf/docker-compose.yml run --rm ros2-perf
```

The T2S network simulator probe/input-preparation runner is available now:

```bash
python3 -m scripts.run_t2s_network_sim
python3 -m scripts.run_t2s_network_sim --prepare-inputs
python3 -m scripts.run_t2s_network_sim --replay-inputs --data-rate-mbps 10
python3 -m scripts.run_t2s_network_sim --json
```

On machines without ns-3 or OMNeT++, this runner still prepares CSV traces and
reports the missing tools explicitly.

The T2E Docker/netem runner is available now:

```bash
python3 -m scripts.run_t2e_netem --prepare-inputs
python3 -m scripts.run_t2e_netem --scenario wifi_loss_jitter --run --analyze
```

It prepares CSV traces on any machine. Running the Docker emulation requires a
running Docker daemon and container `NET_ADMIN` capability so `tc netem` can be
applied inside the sender container.

The T2E ROS 2/netem runner is also available:

```bash
python3 -m scripts.run_t2e_ros2_netem --dry-run
python3 -m scripts.run_t2e_ros2_netem --scenario wifi_loss_jitter --rmw rmw_fastrtps_cpp --run --analyze
python3 -m scripts.run_t2e_ros2_netem \
  --all-rmws \
  --components control,state,sensor,debug \
  --repeat 3 \
  --runtime-s 30 \
  --run-id baseline_v1 \
  --run --analyze
```

This runs real ROS 2 `performance_test` publisher/subscriber containers under
Docker/netem and parses the subscriber CSV log. The matrix mode writes JSONL
metrics plus a summary ranking under `results_t2e_ros2/`.

Generate a human-readable report after a run:

```bash
python3 -m scripts.report_t2e_results \
  --metrics results_t2e_ros2/baseline_wifi_v1_metrics.jsonl \
  --summary results_t2e_ros2/baseline_wifi_v1_summary.json \
  --markdown results_t2e_ros2/baseline_wifi_v1_report.md \
  --csv results_t2e_ros2/baseline_wifi_v1_report.csv \
  --title "Baseline Wi-Fi v1"
```

For `rmw_zenoh_cpp`, the runner defaults to a router topology. It starts a
third `rmw_zenohd` container and points publisher/subscriber sessions at
`tcp/zenoh-router:7447` with `ZENOH_SESSION_CONFIG_URI`. This avoids relying on
multicast scouting or peer loopback advertisements inside Docker bridge
networks. Use `--zenoh-topology peer` only when explicitly testing peer mode.

A lightweight trace replay is available for fast sanity checks before ns-3:

```bash
python3 -m scripts.export_traces --scenario warehouse_50_constrained --format csv
python3 -m scripts.replay_trace traces/warehouse_50_constrained.csv \
  --data-rate-mbps 20 \
  --base-delay-ms 5 \
  --jitter-ms 2 \
  --loss 0.01
```

To run the ns-3 starter in an external ns-3 workspace:

```bash
export NS3_WORKSPACE=/path/to/ns-3
python3 -m scripts.run_ns3_replay \
  --trace traces/warehouse_50_constrained.csv \
  --data-rate 54Mbps \
  --delay 2ms
```

Add simulator-specific importers after the manifest is stable:

```text
scripts/export_traces.py
  -> traces/fleetqox_workload.jsonl

external/ns3/
  -> consumes traces
  -> emits results/ns3_*.jsonl

external/omnetpp/
  -> consumes traces
  -> emits results/omnetpp_*.jsonl
```

The first version should be trace-driven. It should not try to run ROS 2 inside
ns-3 or OMNeT++.

The trace schema should include:

- timestamp;
- flow id;
- source robot;
- destination class;
- size;
- deadline;
- lifespan;
- flow class;
- selected policy;
- degradation state;
- semantic utility.

```bash

sudo apt install ros-jazzy-performance-test
sudo apt install iproute2
```

For container/network testbeds:

```bash
docker --version
tc -V
```

For full robot simulation:

```bash
ros2 --version
gz sim --versions
```

The exact package names may need adjustment by ROS distribution.
